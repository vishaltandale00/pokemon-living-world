# Emergent-Identity Engine — Architecture & Phased Build Plan

*LLM-authored persistent structures over a deterministic Pokémon substrate. Plan only. All line refs verified against the live tree (read 2026-06-17).*

## Vision in one paragraph

Today the world can only speak six verbs (`Development` union, `types.ts:166-172`), only know four factions and eleven roles (`types.ts:7,16-27`), and only advance a hardcoded five-chapter spine whose win conditions are compiled closures (`story.ts:20-21,26-103`). The Emergent-Identity Engine lifts the LLM from *dialogue/meter author* to *runtime dungeon-master/author of structure itself*: at runtime it can mint factions, roles, slots, objectives, buildings, rivalries, and bespoke dialogue options as first-class persistent world objects, so "start a rival Pokémon gang as a startup and become its kingpin" is a real, trackable, winnable arc that we never hardcoded. Two playthroughs diverge permanently because the *content* is generated, yet the *mechanics* are not: every invented goal is expressed as a small list of **typed, sim-checkable predicates** (`rep>=X`, `badges>=N`, `defeated(npcId)`, `controls(buildingId)`, …) that the deterministic sim — not the LLM — evaluates. The four non-negotiables hold by construction. **NN1 (sim is truth):** every new mutation still flows through the single `applyProposal` → `applyDevelopment` clamp/try-catch/log/save chokepoint (`store.ts:115-181`); the only payoffs are a closed, clamped `Effect` union. **NN2 (LLM proposes, sim disposes — now over structure):** the LLM emits open verbs + predicates; the sim validates, drops malformed proposals silently (`store.ts:121,179`), and re-checks every gate against real state at claim time. **NN3 (offline always works):** the seed factions/slots/chapters become the initial registry rows = the authored scaffold, and a generalized `heuristicTick` advances any predicate-backed objective with no key — emergent identity is a *with-key superpower*, never a dependency. **NN4 (real-Pokémon feel):** no `Effect` can mint money/badges/XP; those stay in the battle economy (`BattleScene.ts:294-300`), so every combat gate is still earned through a real, no-one-shot battle.

## The model — generic entity/structure store + coexistence

The containers in `WorldState` (`types.ts:137-148`) are *already* generic `Record<string, …>` dictionaries; only the value-types and the mutation verbs are closed. We keep every existing typed collection (`npcs/slots/towns/buildings`) **byte-for-byte as-is** and add open registries beside them:

```
interface WorldState {
  // ...unchanged: day, player, npcs, slots, towns, buildings, events, rumors, pendingOffers, dialogueCache...
  factions?:   Record<string, { id; label; repAxis; foundedDay; foundedBy }>;  // NEW open registry
  roles?:      Record<string, { id; label }>;                                  // NEW open registry
  objectives?: Record<string, Objective>;   // NEW — replaces the implicit STORY[] index
  entities?:   Record<string, Entity>;       // NEW — orgs / rivalries / locations / custom options
  edges?:      Edge[];                        // NEW — typed relations across the UNIFIED id-space
}
```

- **`Entity`** is the one open shape for invented structures: `{ id, kind:string, label≤60, blurb≤240, state:'active'|'dormant'|'resolved'|'defunct', createdBy:'seed'|'director'|'player', pinned?, …typed-but-optional payloads keyed off kind }`. `kind` is an **open string** (seed-known: `faction|role|objective|location|rivalry|option|org`); the LLM may coin more.
- **`Edge`** `{ from, to, kind:string, day, weight? }` is a flat array over a **unified id-space** — `from`/`to` may be any entity, npc, slot, building, town id, or `'player'`. This is how an invented org attaches to the seed world (`controls`→`rocket_warehouse`, `member_of`←`james`) without changing one seed type. Flat array → `JSON.stringify` stays trivial and old-save migration is a no-op.
- **Open the closed enums:** widen `FactionId`/`RoleId`/`Building['kind']` (`types.ts:7,16-27,89`) to `string`; make `Reputation` a `Record<string, number>` (`addRep` at `store.ts:65-74` already iterates `Object.entries`, so adding an axis becomes data, not a type change — it just auto-inits a missing axis to 0).

**Coexistence is the spine-safety guarantee.** The seed's five slots (`seed.ts:64-75`), four factions, and five chapters are migrated into these registries as `createdBy:'seed', pinned:true` rows — they *demote to seed examples + the offline scaffold* exactly as the North Star mandates, and become structurally immortal (no verb can delete or rewrite a pinned row). `load()` (`store.ts:18`) gains a ~3-line migration defaulting the new collections empty and mapping the old `player.story` int (`types.ts:123`) onto which seed objective rows are `resolved`. The few code-sites that read seed ids structurally (`director.ts:206,215`; `story.ts:148-149`) become registry/graph reads. **Hard caps** keep the single stringified blob (`store.ts:49`, ~5 MB localStorage) bounded: entities ≤120, edges ≤600, factions ≤16, options ≤8/npc, active objectives ≤24; tick-time pruning reaps non-pinned `resolved/defunct/dormant` rows and compacts 30-day-old resolved objectives into one `WorldEvent` (the *fact* survives in the 400-capped log even after the *structure* is reaped).

## The predicate DSL — the crux

A small **closed union of operators** (safe to fix — this is the trusted substrate) parameterized by **open string ids**. The LLM writes the fiction (title/blurb/why); it emits the *mechanics* as `Predicate[]` (requirements / objective completion) and `Effect[]` (grants / payoffs). New types in `types.ts`:

```
type Predicate =
  | { t:'rep';        axis:string;  gte:number }       // player.reputation[axis] >= gte
  | { t:'badges';     gte:number }                      // player.badges >= gte
  | { t:'defeated';   npcId:string }                    // flags['beat_'+npcId] (set at BattleScene.ts:291)
  | { t:'controls';   buildingId:string; by?:string }   // buildings[id].owner === (by ?? 'player')
  | { t:'day';        gte:number }                      // state.day >= gte
  | { t:'hasItem';    id:string;  gte:number }          // player.items[id] >= gte
  | { t:'holdsRole';  slotId:string }                   // slots[slotId].holder === 'player'
  | { t:'flag';       key:string;  value?:boolean }     // flags[key] === (value ?? true)
  | { t:'invitation'; slotId?:string };                 // a live pendingOffer exists for this slot

type Effect =                                            // the ONLY payoffs the sim will apply
  | { t:'addRep';      axis:string; amount:number }      // → addRep, clamp |amount|<=20, axis auto-created
  | { t:'setFlag';     key:string;  value?:boolean }
  | { t:'adjustMeter'; town:string; meter:'prosperity'|'rocketInfluence'; delta:number } // |delta|<=15→0..100
  | { t:'addItem';     id:string;   amount:number }      // |amount|<=10, count>=0
  | { t:'setOwner';    buildingId:string; owner:string|null }  // the only legit path to controls()
  | { t:'setHolder';   slotId:string; holder:string }    // routed through the claimSlot deposition path
  | { t:'rumor';       text:string };                    // cosmetic, length-capped
```

**One evaluator, default-deny.** `evalPredicate(state, p): {ok, why}` is a pure read; unknown `t` → `{ok:false}` so a malformed predicate can never auto-pass (anti-god-mode by default). `meetsRequirements` (`store.ts:77-92`) is rewritten as a fold over `slot.requires: Predicate[]`, **keeping its exact `{ok, missing[]}` contract** so the HUD and `claimSlot` keep working. The static/event split is preserved: `claimSlot` (`store.ts:94-112`) treats `defeated` as claim-time (the battle already set `flags['beat_<holder>']` at `BattleScene.ts:291`), so combat gates are only satisfiable through a real battle. **Deliberate expressivity limit:** a `Predicate[]` is pure AND. Disjunction lives at the slot/objective layer (two slots, two objectives), never as a nested boolean tree — that is what keeps every goal sim-checkable and exploit-auditable, and keeps the `missing[]` list legible. `RoleSlot.requires` (`types.ts:36-41`) becomes `Predicate[]`; `RoleSlot` gains `grants?: Effect[]`, `protected?: boolean`, and `claimVia?: 'static'|'battle'` (replacing the `defeatHolder` boolean). A thin compat adapter converts the legacy `{minRep, badges, defeatHolder, invitation}` struct → `Predicate[]` at load so seed data and the battle bridge keep passing. The crux holds: **the LLM may invent any goal and any reward, but the reward only lands once the predicate-defined work is actually done in the sim**, because the only paths that improve player standing are `claimSlot` (re-checks `meetsRequirements`) and `Effect` application (clamped, no `addMoney`/`addBadge`/absolute-set arms exist).

## Proposes/disposes — generic ops + universal invariants

`applyDevelopment`'s `switch` (`store.ts:134-181`) becomes a **registry** `OPS: Record<string, handler>`, and `DIRECTOR_SCHEMA.kind.enum` (`director.ts:20`) is *generated from `Object.keys(OPS)`* — eliminating the three-place union duplication that is the current #1 blocker. The six legacy arms move into behavior-preserving adapter handlers (so seed/heuristic/battle keep emitting them verbatim at every commit). New open verbs:

- **`create_entity`** (`entity: faction|role|slot|objective|building`) — mints a registry row / `RoleSlot{requires:Predicate[], grants:Effect[]}` / `Objective{done:Predicate[], onComplete:Effect[]}`; `building` reuses `findBuildingSpot` (`store.ts:211`) verbatim.
- **`update_entity`** — **whitelisted** patch only (npc attitude/personality, town mood/meters, slot *title only*, objective hint/active). It is explicitly forbidden from writing `holder`, `requires`, `badges`, `reputation`, or `owner='player'` — those are *earned*, never patched.
- **`link_entities` / `offer` / `grant` / `add_option` / `retire_entity`** — relations, the generalized `role_offer` over generated slots, the earnability chokepoint, bespoke dialogue choices, and the generalized `vacate_slot`.

**Universal invariants, all enforced at the handler boundary (violation → drop the op via the existing `store.ts:121` try/catch, never a throw, never a partial mutation — validate-before-mutate ordering):**

1. **No dangling refs / unified-unique ids.** Every id a handler reads is checked against its registry first (a single `resolveId(id)` gate generalizes today's `if (!t) return null` / `if (!slot||!from) return null` guards at `store.ts:138,154`). Every created id is unique across `npcs ∪ slots ∪ towns ∪ buildings ∪ entities`, so an edge can never alias a seed object. `create_edge` requires both endpoints to resolve; pruning cascade-deletes dangling edges.
2. **Earnability / anti-god-mode (the headline guarantee, three independent layers).** (a) Compiler/director soft gate. (b) Sim hard gate: a minted objective's `done` must contain ≥1 currently-FALSE predicate (no no-op win), every minted power slot must carry ≥1 non-trivial `requires` predicate (no empty-requirement god-slot), and `grant`/`onComplete` re-evaluate the authorizing predicates *at apply time* and drop if unmet. (c) Substrate backstop: `Effect` is a closed, individually-clamped allowlist with no money/badge/absolute-rep arm — so even a fooled gate yields only bounded soft-rep. The single legitimate path to `controls(buildingId)` is a `setOwner` effect gated by a battle/objective.
3. **No contradictions / idempotence.** Ids are slugged + collision-checked → `create_entity` rejects on collision (mutation requires explicit `update_entity`); `offer` keeps its dup-guard (`store.ts:155`); a slot's `holder` is only ever written by `claimSlot`/gated-`grant`/`retire`.
4. **Append-only history.** Every successful op `logEvent`s (`store.ts:56`) with kind + reason + affected ids; the 400-cap slice (`store.ts:58`) is the only log mutation. Corrections are new events, never edits — world memory is monotonic and reconstructable.
5. **Footholds / rate limits.** The `applyProposal` slice-to-4 (`store.ts:117`) gains a per-kind budget (≤2 `create_entity`/tick, ≤1 new faction/tick, via a small `proposalLedger`). Founding a faction requires a foothold predicate to already hold, else it is created **dormant** (registered repAxis only, owns nothing, can gate nothing) until it earns a presence — so "start a rival gang" is a multi-day arc, not a free button.
6. **Spine-safety.** `retire_entity`/`update_entity{active:false}` reject any target referenced by an active objective's `done`/`requires` predicates or by a pinned `blocks`/`succeeds` edge (now mechanically detectable because relations and win-conditions are *data*, not closures), and reject if the op would leave zero satisfiable active objectives.

## Free-form intent + Director-as-author

**Player goals enter as text** through a new `'J'` ("declare ambition") hotkey opening a DOM `<textarea>` overlay (reusing the existing Settings DOM-overlay pattern, 240-char cap), plus a universal *"[Tell them what you really want…]"* dialogue choice appended in `drawDialogue` so ambitions can be declared in-character (that NPC becomes the natural `fromNpc` for a spawned offer). Raw text mutates **nothing**: it is pushed to an `ambitionLog` (audit trail) and handed to a *separate* second LLM call — the **intent-compiler** (`src/llm/intent.ts`, mirroring `DIRECTOR_SCHEMA`'s flat/nullable style so the same `chatJSON` path works). The compiler embeds the same `worldDigest()` so it can only reference real ids, and returns `{ feasible, rejectReason, objective:{title,hint,directive, done:Predicate[], onComplete:Effect[]}, scaffolding:Development[], supersedes }` — fiction + mechanics, never free-string win conditions. `store.installAmbition` is the single dispose chokepoint: it runs the same anti-god-mode hard gate, installs `scaffolding` through `applyProposal` (same clamp/try-catch), writes the `Objective`, and either sets it as the active thread or queues it behind the spine.

**The active objective generalizes.** `Objective { id, source:'spine'|'player', title, hint, directive, done:Predicate[], onComplete:Effect[], status, order }` replaces the `STORY[]` closures (`story.ts:20-21,26-103`). `currentChapter` → `activeObjective(w) = w.objectives[w.activeObjectiveId]`; `advanceStory` (`story.ts:143-169`) → `advanceObjectives`, folding `evalPredicate` over each `done[]` and firing `onComplete` `Effect[]` (keeping the existing guard loop and the champion auto-claim's intent via `holdsRole`). The Director's digest thread block (`director.ts:69-74`) reads the active objective and **prints its UNMET predicates** so the model knows exactly what structure to grow toward ("UNMET: control rocket_warehouse; ember reputation ≥50 (have 0)"). One added system clause makes the Director the goal's co-author: *grow obstacles/rivals/offers toward the unmet predicates, but never set a predicate true yourself.*

**Offline behavior is first-class.** With no key the affordance still opens but offers a **curated menu** of 3–4 pre-compiled, sim-validated `SEED_AMBITIONS` (which double as few-shot examples in the compiler prompt) instead of free typing — we never silently no-op. The authored five-chapter spine remains the default active objective. The generalized `heuristicTick` (`director.ts:143-226`) replaces its `switch(ch?.id)` literal-chapter branch with a generic loop that nudges the meters/offers each *active objective's unmet numeric predicates* reference (it already does this per-chapter — we generalize the pattern), so a persisted LLM-authored objective keeps advancing keyless. Seed objectives are simply the pinned baseline rows of the same loop.

## Validating a generative game

A property-based **Generative Validation Harness** (`src/eval/`) extends `docs/INTENT_AND_EVAL.md` from "score the fixed seed/spine" to "score the generative space": the existing S1–S9 become the **always-green deterministic floor**; randomized ambitions are the **dynamic ceiling**. It runs in the browser sim across parallel `WorldStore` instances (per-instance `SAVE_KEY` suffix so saves don't collide), driven by a real key under a shared token bucket. The core idea: an **`ActorPolicy`** is a deterministic greedy *predicate-solver* — given any goal expressed as `Predicate[]`, it chases the cheapest unmet predicate via a fixed action→predicate table (`badges`→battle a gym, `defeated`→battle that NPC, `controls`→satisfy the ownership precondition, `invitation`→drive the gating rep then accept via `acceptsOffer`→`claimSlot`). Because it only ever calls the *same chokepoints a human uses* (`BattleScene.onWin`→`claimSlot` at `BattleScene.ts:309-313`; `WorldScene.pickChoice`→`addRep`/`claimSlot` at `WorldScene.ts:554-571`), the harness can prove **NO-GODMODE**: if the policy reaches the goal, it did so only through sim-validated paths.

Per randomized ambition (e.g. "found the Ember Syndicate and become its kingpin", seeded RNG → reproducible) it asserts five **arc-shape properties** over the run trace plus the four non-negotiables as **per-tick hard gates**:

- **VIABLE** — every minted predicate references a live id and is policy-drivable (orphan predicate / dead goal → fail at tick 0).
- **COHESIVE** — each tick's developments touch the ambition's town/NPCs/faction (generalizes the "on-thread" check); LLM-judge cohesion ≥4, cross-checked against the event log so the judge can't hallucinate a contradiction.
- **PERSISTENT** — minted structures survive ticks and a `JSON.stringify`→parse round-trip (no silent GC, no holder reset).
- **EARNABLE** — every grant has its corresponding chokepoint event in the log; rep moved only in clamped steps; badges only via real gym wins.
- **WINNABLE** — ≥90% of VIABLE ambitions reach `won` within the tick cap; **0% softlock** (a softlock = active objective with an unmet predicate and *no* legal action — proven by enumerating the action table, distinguishing "engine softlocked" from "policy too dumb" → the latter is logged `abandoned`, excluded from the denominator). **NN3 replay:** every WON ambition's objective is re-run through the keyless `heuristicTick` and must still reach `done` — surfacing any predicate kind the generic heuristic can't nudge as a *real red finding*, not noise. Failing seeds **shrink** (bisect to the first violating tick → minimal repro), QuickCheck-style. The harness adds **zero production code** — it only reads the generalized engine and imports the live clamp constants (`store.ts:69,139-147`) as the single source of truth so the test can't drift from the code.

This deliberately exercises every subsystem at minimum width.

## Phased rollout beyond the slice

Each phase is independently shippable and keeps the seed game playable at every commit (seed data migrates into registries rather than being removed). Recommended order — Step A is the crux and is independently testable against existing seed slots before any open-union work.

- **P0 — Predicate vocab + evaluator (Step A).** Types, `evalPredicate`/`applyEffect`, `meetsRequirements` fold, seed-slot migration, load adapter. *Accept:* all 5 seed slots claimable exactly as before; unit-test the evaluator against seed state.
- **P1 — Op registry + open Development union (Step B).** `OPS` registry, legacy arms → adapters, schema generated from registry keys. *Accept:* pure refactor; game plays identically.
- **P2 — Open enums + registries (Step C).** Widen `FactionId`/`RoleId`/`Building['kind']` to string; `factions`/`roles` registries + seed rows; swap the two `BattleScene` data-lookups (badge id check `BattleScene.ts:292`, faction→axis map `:303`). *Accept:* a new faction earns rep on defeat.
- **P3 — Data-driven objectives + generic heuristic (Step D).** `STORY[]` → seeded `objectives` rows; `advanceStory`→`advanceObjectives`; `currentChapter`→`activeObjective`; `heuristicTick` generic over unmet predicates; migrate `player.story`. *Accept:* keyless 5-chapter spine completes end-to-end; **highest-value step** (offline scaffold becomes data).
- **P4 — Entity graph + edges + full create_* family (Step E substrate).** `entities`/`edges`, `entityGraph.ts` query helpers, `create_entity`/`create_edge`/`link_entities`/`retire_entity`, foothold/rate-limit budgets, pruning + compaction, `controls` via `setOwner`. *Accept:* the Ember Syndicate org persists/diverges; save size stays bounded; collisions/dangling edges silent-drop.
- **P5 — Free-text compiler + custom options + spine-safety.** Full `compileAmbition`, the in-character dialogue choice, `add_option` rendered in `pickChoice` (use-time gated), the destructive-verb dependency scan, journal UI for queued objectives. *Accept:* superseding then abandoning an ambition cleanly resumes the spine; god-mode requests rejected.
- **P6 — Generative Validation Harness.** Invariant kernel → `ActorPolicy` predicate-solver → randomized ambitions over the 5 arc-shape properties + NN3 keyless replay → parallel fleet + LLM judge + seed-shrinking + CI gate (overall ≥0.80, no arc-property <0.65, zero NO-GODMODE/NO-SOFTLOCK/G-SIM/G-FALLBACK violations).

## Risks & open decisions for the user

1. **Two LLM calls per beat, or one?** The plan uses a *separate* intent-compiler (`intent.ts`) distinct from the nightly Director. **Recommended default: keep them separate.** The compiler is synchronous/interactive (player is waiting, needs tight validation + reject reasons) while the Director is the async nightly tick; merging them couples latency budgets and muddies the system prompts. Cost is one extra call only when a player actually declares an ambition (rate-limited to 1/in-game-day).

2. **Disjunction expressivity — flat AND lists, or nested boolean trees?** The plan deliberately forbids nested predicates (OR lives at the slot/objective layer). **Recommended default: keep the flat-AND limit.** It is the single property that makes goals exploit-auditable and the `missing[]` HUD legible; the cost is that "rep≥40 OR control-the-warehouse" must be authored as two parallel objectives, which is acceptable and arguably clearer to the player. Revisit only if real ambitions feel cramped.

3. **How aggressively to bound generated structure?** Caps (entities ≤120, factions ≤16, etc.) + pruning + 30-day compaction protect the ~5 MB single-blob save, but over-pruning could reap an org the player still cares about. **Recommended default: ship the caps but pin anything referenced by an active objective, a pending offer, or a `member_of`/`controls` edge to `'player'`** — only truly orphaned non-active rows are reaped. Expose the numbers as constants so they're tunable from harness soak-test data, not guesses.

4. **Should a player ambition be able to *supersede* the authored spine, or only run parallel to it?** Superseding makes player intent feel powerful but risks the authored Giovanni arc feeling abandoned. **Recommended default: supersede-by-queue, never delete** — the displaced spine objective goes `status:'queued'` and auto-resumes when the ambition resolves or is abandoned, with a journal entry to re-activate it manually. This preserves NN3 (the spine is always reachable) while honoring player agency.
