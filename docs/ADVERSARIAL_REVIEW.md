# Adversarial Review — Emergent-Identity / Generative-Kernel Vision

## The 5 most dangerous attacks — ranked by severity × likelihood

**1. The entire two-layer engine is vaporware; the "guarantees-by-construction" are unfalsifiable. — FATAL.** I verified this directly: `src/eval/`, `interpreter.ts`, and `intent.ts` do not exist, no planned symbol (`evalPredicate`, `runRules`, `ActionDef`, `predicate_cross`, `ruleState`, `firedThisRun`) appears anywhere in `src/`, there is no test script and no test file. The shipped `applyDevelopment` is a fixed 6-case `switch` (store.ts:135-175) over a closed reducer — the "generic state-machine kernel" the whole vision rests on has never compiled. Every load-bearing property ("provably terminates," "NO-GODMODE," "deterministic," "0% softlock") is prose describing software that does not exist. You cannot call a thing "exploit-proof by construction" when the construction, the prover, and every proof obligation are unwritten. This doesn't kill the *idea*, but it kills the claim that the idea is *de-risked*: the docs present a research bet as a verified architecture.

**2. The "sim-trusted deterministic kernel" is already non-deterministic in the code it's built on. — FATAL (as stated), MITIGABLE (mechanically).** Confirmed at store.ts:157 (`offer_${Date.now()}`), 190 (`bld_${...}_${Date.now()}`), 203/215-216 (`Math.random()` for target selection and placement) — and these ids/coords are *persisted into the save*. "Same save + same op stream ⇒ byte-identical post-state" (LAYER.md:179), the literal foundation of the eval harness's replay and seed-shrinking, is false on tick 0 of the existing game before any LLM rule is added. The fix (seeded PRNG + day+seq counter) is real and bounded, but two things make this damning rather than trivial: (a) the team already *asserted* this determinism in three docs while the code violated it — their own verification missed it once, so "we'll just audit it" has a credibility problem; (b) the harness that is supposed to *prove* coherence is the same harness whose minimal-repros are unsound until this is fixed. Severity is fatal *to the claim as written*; the underlying property is recoverable.

**3. Open value/goal dimensions make anti-cheat irrelevant: the kernel guards old primitives while the LLM mints unbounded new currency that gates self-authored wins. — FATAL.** The `|20|` rep clamp is per-fire, never aggregate, and `addRep` auto-creates any axis (store.ts:65-69 iterates an open `Record<string,number>`). An authored `day_tick` rule pumping a new `ember` axis behind a cheaply-true guard mints standing autonomously with zero battles, and an authored Objective `done:[rep(ember,gte:50), entityField(heist,phase,eq:pulled)]` — the docs' own Heist example — is a self-authored victory loop with no combat gate. The anti-god-mode check ("≥1 currently-FALSE predicate") waves it through. The kernel's determinism/no-mint/save-integrity guarantees remain perfectly intact *and perfectly irrelevant*, because difficulty and stakes were never kernel properties. This is the deepest finding: the anti-exploit story protects the wrong surface. It forces a redesign of what the kernel anchors, not just how.

**4. The LLM cannot author the one mechanic that makes Pokémon fun (a balanced battle), and every "new mechanic" bottoms out in soft-rep/meter/flag flavor. — FATAL to the value prop.** Combat difficulty lives in hand-tuned `makeMonster(species, level)` calls and compressed type-effectiveness constants (monsters.ts) — none of it is in the Predicate/Effect/Schema vocabulary, and the docs forbid minting XP/levels. So either invented goals reuse the same five seed battles (divergence collapses to flavor text) or, if `create_entity{npc}` is added, the LLM authors free-win or impossible-wall fights with no level-curve logic. Meanwhile the worked example's *entire* payoff for a six-day arc + a real Archer fight is `+8` on an invisible meter (LAYER.md:224-228) — anticlimax by construction, in the authors' own showcase. The vision wants "two playthroughs diverge completely" *and* "feels like a real Pokémon game"; the architecture delivers a trade-off where the divergent content is structurally thinner than the hand-authored content (10 personality trees, 5-chapter spine) it replaces. The more divergent a playthrough, the less of the actual fun it contains.

**5. Silent-drop-only validation + default-deny predicates create undebuggable, undiagnosable softlocks with no feedback loop and no tooling. — MAJOR.** Confirmed: the only validation failure mode is `catch { /* skip silently */ }` (store.ts:121). A typo'd field (`phse` vs `phase`) is structurally valid JSON that evaluates false forever — indistinguishable from "not yet satisfied" — so an objective installs clean and is permanently unwinnable with no in-game signal. The nightly Director gets *no* `rejectReason` feedback, so it re-emits the same dead construct every night, burning author budget on no-ops. There is no trace viewer, no rule-dependency graph, the event log is FIFO-capped at 400, drops aren't logged at all, and the one named defense (the WINNABLE harness's action-table enumeration) is blind to cross-tick predicate oscillation *and* doesn't exist. Combined with non-deterministic ids (#2), a failing save can't even be reliably re-run. This is MAJOR not FATAL because it's addressable with real tooling investment — but that tooling is in no phase and isn't budgeted.

## What actually survives

- **The proposes/disposes split is sound and worth keeping.** "LLM emits data, sim validates and applies through one chokepoint" is the right shape. The shipped `applyProposal`/`applyDevelopment` already demonstrates it works for a closed verb set. The discipline of *all authored content reduces to clamped primitives* is genuinely a good safety primitive.
- **The hard anti-mint guarantee on money/badges/XP is real and solid.** There is no Effect arm that mints currency or levels; battles are the only source of league rep/badges (BattleScene.ts:298,308). That part of "exploit-proof" holds — the leak is *rep as a soft currency that hard gates depend on*, not the hard primitives themselves.
- **Determinism is achievable.** Once `Date.now()`/`Math.random()` are replaced with a seeded counter+PRNG, byte-identical replay is a real, attainable property. The bug is shipped, not fundamental.
- **Data-driven objectives (Predicate[]) are a good refactor independent of the rest.** Replacing the `STORY[]` closures with typed predicates is valuable cleanup that improves the *existing* game regardless of whether Layer 2 ever ships.
- **The seed game is already the product.** 793 lines deliver the full advertised arc with real battles, four-axis rep, two alternate endings, and a keyless offline path. There's a `playthrough.gif`. It works today.

## What must change

1. **Bound the kernel to mechanic-TEMPLATES, not a free rule-language.** The free `WHEN→THEN` rule web is where every fatal interaction lives (cross-tick oscillation, budget death, non-confluence, undebuggable softlocks). Instead, ship a small fixed catalog of *parameterized* mechanic templates (a "heat/stealth" template, a "faction-control" template, a "crew-buildup" template) that the LLM *instantiates with parameters*, not *authors from primitives*. A template is human-authored, human-tested, terminating, and legible; the LLM only fills slots. This keeps divergence (different parameters, different fiction) while killing the Turing-tarpit. This is the single highest-leverage de-scope.

2. **Anchor every authored win and every standing grant to a protected primitive.** New rep axes must be *redistributive* (a conversion of already-earned battle rep) or capped by a global per-axis velocity bound enforced *inside* `addRep`, not in a validator the data flows around. Any authored Objective treated as a win must include ≥1 `defeated()`/`badges` predicate against a *seed* NPC (not a self-authored pushover). Yes, this re-couples emergent identity to combat — that coupling is the price of "exploit-proof," and the docs can't have both.

3. **Put a human/eval gate on balance and fun, and stop pretending validity is fun.** The harness measures VIABLE/COHESIVE/PERSISTENT/EARNABLE/WINNABLE — all validity. None measures whether the payoff was worth the grind. A green harness on a `+8`-ember six-day arc is a passing test of a boring game. Either add human playtest gates or accept that the autonomous improvement loop is blind to its own goal and will halt at "maximally valid, possibly joyless."

4. **Fix determinism first, before any engine work.** Seed the PRNG, replace wall-clock ids with day+seq, add a replay test. This is a prerequisite for *any* of the safety claims and is a day or two of work.

5. **Build the debugger before the language.** If you ship a programming language authored at runtime, you owe it a trace + dangling-ref/dead-rule/unsatisfiable-objective inspector and a *drop-reason feedback channel* into the next Director digest. Without it, every divergent save is a uniquely-broken JSON swamp no human can diagnose.

6. **Cap generativity and precompute the frontier.** Make the intent-compiler async with a curated `SEED_AMBITIONS` fallback, pre-materialize one frontier step during the nightly tick (when the player already waits), and render authored verbs from their typed `do[]`/`requires[]` *without* an LLM call. Accept that the keyless default path is a curated menu, and be honest in the eval that this is "most users."

## The riskiest assumption & the cheapest spike

**The one assumption everything rests on:** *an LLM can author a coherent, balanced, FUN, bug-free new mechanic — as composed data — that a player can understand and that doesn't softlock, over a long horizon.* Every fatal finding is a corollary of this being false. Note it is **not** "can the kernel be made deterministic" (that's mechanical) — it's whether the *generative content itself* clears the fun/coherence bar the kernel can't enforce.

**The cheapest spike (≤1 day, zero engine):** Hand-write the Predicate/Effect/Rule/Action/Objective JSON schema as a *paper spec* (no interpreter). Prompt the actual target model, 15 times, with the real `worldDigest` + a one-line ambition ("run a heist on the Pewter warehouse"), asking it to emit the full schema+rule+action+objective bundle. Then **hand-simulate** each of the 15 bundles on paper for 6 ticks and score three things: (a) does it parse and reference only live ids/fields (coherence)? (b) does the objective become satisfiable *and* require a real battle, without minting unbounded rep (balance/no-exploit)? (c) would a player understand what the verb does from its data alone (legibility)? If fewer than ~12/15 pass all three *with no interpreter bugs to blame*, the generative vision is dead before you write `interpreter.ts` — and you've spent a day instead of months. This directly tests the assumption the entire P0–P6 + P-L0–P-L5 build presupposes.

---

## Honest verdict

**Spike-first, then build a bounded version — do NOT build the full free-rule-language vision.**

The shipped 793-line authored game is already the product; the multi-month VM-plus-compiler-plus-harness build, by its own worked example, spends the hardest category of games engineering to make that fun game slower, less coherent, API-dependent, and topped with a stat-tracker — while leaving the genuinely cheap, high-leverage wins (more towns, more species, better authored dialogue grounding) unbuilt. The free rule-language is a Turing-tarpit whose fatal failures (unbounded self-minted currency gating self-authored wins, cross-tick softlocks, undebuggable divergent saves) are structural, not bugs. **The single sentence a skeptical co-founder needs to hear:** *Spend one day proving an LLM can author one coherent, balanced, legible mechanic on paper before you spend three months building the engine that assumes it can — because the engine is provably correct and the content is unprovably fun, and you're about to build the part that already works.*
